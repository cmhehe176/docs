Dưới đây là mô tả chi tiết cách dữ liệu luân chuyển trong ứng dụng theo từng bước.

🔷 1. Các thành phần trong hệ thống ​

Ứng dụng được chia thành các Layer chính:

  • Presentation Layer (Giao tiếp API) – Controllers
  • Application Layer (Xử lý nghiệp vụ, CQRS) – Command/Query Handlers, Services
  • Domain Layer (Định nghĩa logic nghiệp vụ) – Entities, Repository Interface
  • Infrastructure Layer (Tương tác Database) – Repository Implementation, ORM

🔷 2. Luồng dữ liệu với Command (Ghi - Write) ​

Ví dụ: Người dùng gửi request để tạo tài khoản mới.

🔽 Luồng Request khi tạo User (POST /users) ​

  1. Client gửi POST /users với dữ liệu { "name": "John Doe", "email": "john@example.com" }.

  2. Controller (UserController) nhận request và gọi CommandBus để thực thi CreateUserCommand.

  3. CommandBus chuyển command này tới CreateUserHandler (Application Layer).

  4. CreateUserHandler:

    • Tạo một User Entity trong Domain Layer.
    • Gọi UserRepository để lưu vào database.

  5. UserRepositoryImpl (Infrastructure Layer) sử dụng ORM (Prisma, TypeORM, MongoDB, etc.) để lưu User vào DB.

  6. Trả về kết quả cho CommandBus → Controller → Client.

🔁 Luồng tổng quan (Write Path) ​

zsh
Client → Controller → CommandBus → CommandHandler → Domain Entity → Repository Interface → Repository Implementation → Database

📌 Ví dụ mã nguồn ​

(1) Controller (user.controller.ts) ​

ts
@Controller('users')
export class UserController {
  constructor(private readonly commandBus: CommandBus) {}

  @Post()
  async createUser(@Body() body: { name: string; email: string }) {
    return this.commandBus.execute(new CreateUserCommand(body.name, body.email))
  }
}

(2) Command (create-user.command.ts) ​

ts
export class CreateUserCommand {
  constructor(
    public readonly name: string,
    public readonly email: string,
  ) {}
}

(3) Command Handler (create-user.handler.ts) ​

ts
@CommandHandler(CreateUserCommand)
export class CreateUserHandler implements ICommandHandler<CreateUserCommand> {
  constructor(private readonly userRepository: UserRepository) {}

  async execute(command: CreateUserCommand): Promise<void> {
    const user = new User(Date.now().toString(), command.name, command.email)
    await this.userRepository.save(user)
  }
}

(4) Repository Interface (user.repository.ts) ​

ts
export interface UserRepository {
  findById(id: string): Promise<User | null>
  save(user: User): Promise<void>
}

(5) Repository Implementation (user.repository.impl.ts) ​

ts
@Injectable()
export class UserRepositoryImpl implements UserRepository {
  private users: User[] = []

  async findById(id: string): Promise<User | null> {
    return this.users.find((user) => user.id === id) || null
  }

  async save(user: User): Promise<void> {
    this.users.push(user)
  }
}

🔷 3. Luồng dữ liệu với Query (Đọc - Read) ​

Ví dụ: Người dùng gửi request để lấy thông tin user.

🔽 Luồng Request khi lấy User (GET /users/:id) ​

  1. Client gửi GET /users/123.
  2. Controller (UserController) nhận request và gọi QueryBus để thực thi GetUserQuery.
  3. QueryBus chuyển query này tới GetUserHandler (Application Layer).
  4. GetUserHandler gọi UserRepository để lấy dữ liệu từ DB.
  5. UserRepositoryImpl (Infrastructure Layer) sử dụng ORM để lấy dữ liệu từ database.
  6. Trả về dữ liệu từ QueryBus → Controller → Client.

🔁 Luồng tổng quan (Read Path) ​

zsh
Client → Controller → QueryBus → QueryHandler → Repository Interface → Repository Implementation → Database → Trả dữ liệu về

📌 Ví dụ mã nguồn ​

(1) Controller (user.controller.ts) ​

ts
@Controller('users')
export class UserController {
  constructor(private readonly queryBus: QueryBus) {}

  @Get(':id')
  async getUser(@Param('id') id: string) {
    return this.queryBus.execute(new GetUserQuery(id))
  }
}

(2) Query (get-user.query.ts) ​

ts
export class GetUserQuery {
  constructor(public readonly id: string) {}
}

(3) Query Handler (get-user.handler.ts) ​

ts
@QueryHandler(GetUserQuery)
export class GetUserHandler implements IQueryHandler<GetUserQuery> {
  constructor(private readonly userRepository: UserRepository) {}

  async execute(query: GetUserQuery) {
    return this.userRepository.findById(query.id)
  }
}

🔷 4. So sánh Write Path vs Read Path ​

Hành độngLuồng Command (Ghi - Write)Luồng Query (Đọc - Read)
✅ Bắt đầu từControllerController
✅ Gửi yêu cầu quaCommandBus.execute(command)QueryBus.execute(query)
✅ Handler xử lýCommandHandlerQueryHandler
✅ Thao tác với DBRepository (Save)Repository (Find)
✅ Trả về kết quảvoid hoặc IDUser DTO hoặc null

🔷 5. Tổng kết ​

💡 Với NestJS + CQRS + DDD, dữ liệu di chuyển qua nhiều lớp như sau:

  1. Presentation Layer nhận request (Controller)
  2. Application Layer điều hướng logic (Command/Query Handler)
  3. Domain Layer thực hiện logic nghiệp vụ (Entities, Repository Interface)
  4. Infrastructure Layer thao tác với DB (Repository Implementation)

🎯 Ưu điểm của cách tổ chức này: ✅ Tách biệt Write & Read (CQRS giúp tối ưu hiệu suất). ✅ Dễ dàng thay đổi công nghệ database mà không ảnh hưởng business logic. ✅ Tăng tính maintainable khi phát triển hệ thống lớn.

🚀 Tóm lại, hệ thống NestJS với DDD + CQRS giúp bạn quản lý code chặt chẽ, tách biệt rõ ràng giữa logic business và hạ tầng, đồng thời dễ mở rộng và maintain. 🎯 ​
